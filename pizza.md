# 🍕Пиццерия

Проект, над которым мы работаем - сайт пиццерии на Laravel. 

Наша задача - сделать бекенд для этого сайта, функционал следующий: есть главная страница с товарами - пиццами и напитками. 

Пользователь может добавлять себе в корзину максимум 10 пицц и 20 напитков. После заполнения корзины, пользователь может сделать заказ, 
но для этого ему нужно авторизоваться, а затем заполнить данные для доставки - номер телефона, эл. почта, адрес, время доставки. 

Помимо этого, пользователь может посмотреть список своих предыдущих заказов.

Ещё есть админ. панель, в которой можно добавлять/изменять/удалять товары, просматривать список заказов и изменять их статус 
(например, из статуса “в работе” в статус “доставляется”, после чего этот статус может увидеть у себя пользователь). 

Авторизоваться туда может только человек которому мы лично проставим статус админа в БД.

Требования к реализации:

1. Фреймворк Laravel
2. Проект должен работать докере
3. Должны быть сидеры (seeders), чтобы я мог локально сразу посмотреть на проект с готовыми тестовыми данными
4. Фронт не делать
5. Должны быть тесты. Как минимум по 1 удачному (200) и 1 неудачному (4ХХ) на каждый роут
6. Должны быть минимальные проверки кода на качество (phpstan, php-cs-fixer)

Для тех, кто пришёл прокачать харды и мы заранее их оговорили есть опциональные усложнения:

1. Закешировать информацию о продуктах, время жизни кеша - 1 день. Инвалидировать при обновлении информации. Использовать Redis
2. Асинхронную генерацию отчета через брокер сообщений (получили запрос от юзера, отдали уникальный идентификатор отчета, стали генерить в фоне) через Rabbitmq
 
   Что в отчёте: в отчёте у нас список всех проданных товаров т.е. если велосипед_10 купили 5 раз разные люди, то он 5 раз будет в этом отчёте. Это будет файл .jsonl вида:
   
   ```jsonl
   {"product_name": "велосипед_10", "price": 500, "amount":1, "user": {"id":1}}
   {"product_name": "велосипед_10", "price": 500, "amount":1, "user": {"id":2}}
   ```
3. использовать scheduler для альтернативного запуска генерации отчетов
4. Поиск товаров через opensearch с fuzzy поиском
5. Добавить к продуктам категории и возможность фильтра по ним
6. Бьем приложение на 2 микросервиса. 

    1 - мастер система, которая знает про товары и система для пользователей, 
которая позволяет заказывать эти товары. Общаются эти системы по http и с помощью очередей.

   По http можно запросить у мастер системы какие товары есть, а мастер система, в свою очередь, 
раз в час присылает обновления ассортимента - что то может добавиться, а что то удалиться.  
   
   Для простоты восприятия можно представить, что у нас в компании 2 отдела - первый отдел придумывает рецепты, 
делает им описания и.т.д. у него есть своя админка (но мы её делать не будем), в которой он все это делает и знать не знает кто там и сколько чего заказывает. 

   При обновлении информации он уведомляет об этом отдел заказов. И второй отдел - который заведует именно заказами и доставкой, ведет учёт и.т.д.
